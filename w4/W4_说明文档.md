# S&OP 决策中枢 v4.0 — `w4` 目录说明文档

本文说明 `w4` 包内各文件职责、主要函数/类的参数与行为，以及它们如何对应第四周作业要求，并解释为何采用当前技术选型。

---

## 一、与作业要求的对照关系

| 作业要求 | 在代码中的落点 |
|----------|----------------|
| 废弃 `router.py`，用原生 Agent 路由工具 | 未使用 `sop_hub.router`；使用 `create_tool_calling_agent` + `AgentExecutor`，由大模型根据工具 **description** 决定调用 |
| 检索与表格逻辑封装为独立工具 | `tools.py` 中 `sop_document_search`、`sop_data_analytics`，且 docstring 即工具的精准描述 |
| Tool Calling Agent | `agent.py`：`create_tool_calling_agent` 绑定 LLM 与工具 |
| 多轮对话记忆 | `RunnableWithMessageHistory` + `get_session_history` + `InMemoryChatMessageHistory` |
| Prompt 中带 `chat_history` | `ChatPromptTemplate` 中 `MessagesPlaceholder(variable_name="chat_history", optional=True)` |
| 终端展示工具名与入参、思考链路 | `callbacks.py`：`ColoredToolCallback` 在 `on_tool_start` / `on_tool_end` 打印 Action / Observation |
| 基于 w3 包 | `__init__.py` 将仓库内 `w3/` 置于 `sys.path` 最前，统一 `import sop_hub` |

---

## 二、目录与文件总览

```
w4/
├── __init__.py      # 包初始化：注入 w3 到 sys.path
├── __main__.py      # python -m w4 的入口
├── app.py           # CLI：建索引、建工具、建带记忆的 Agent、交互循环
├── agent.py         # 组装 LLM、提示词、AgentExecutor、RunnableWithMessageHistory
├── tools.py         # @tool 封装：文档检索、CSV 分析
├── callbacks.py     # 彩色终端：工具调用与观测打印
└── W4_说明文档.md   # 本文件
```

## 三、各文件详解

### 3.1 `__init__.py`

**作用**

- 在**首次** `import w4` 或其子模块时执行，把项目中的 **`../w3` 目录**插入 `sys.path` 的**最前面**。
- 保证后续 `from sop_hub.xxx import ...` 解析到 **`w3/sop_hub`**，而不是别处同名包，实现「w4 逻辑基于 w3 交付物」。

**无对外函数**；副作用是修改 `sys.path`。

**为何这样做**

- 不把 `sop_hub` 再复制一份到 `w4`，避免两套代码分叉。
- 不强制 `pip install -e w3`，作业环境只需「仓库根目录 + 虚拟环境」即可运行 `python -m w4`。

---

### 3.2 `__main__.py`

**作用**

- 使 `python -m w4` 等价于调用 `app.main()`。

**函数**

| 名称 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| （模块级） | — | — | `from .app import main`，若 `__name__ == "__main__"` 则 `main()` |

---

### 3.3 `app.py`

**作用**

- **一次性初始化**：读取配置、加载/构建 FAISS、导出 BM25 语料、构造混合检索管线、构造工具列表、构造带会话记忆的 Agent、进入 REPL。
- **多轮会话**：通过固定 `session_id` 与 `RunnableWithMessageHistory` 的 `config` 关联同一条聊天历史。

**函数**

#### `_parse_args() -> argparse.Namespace`

| 项目 | 说明 |
|------|------|
| **作用** | 解析命令行参数。 |
| **参数** | 无（从 `sys.argv` 读取）。 |
| **返回值** | `Namespace`，包含 `dir`, `recursive`, `rebuild`, `session`。 |

支持的 CLI 参数：

| 参数 | 含义 |
|------|------|
| `--dir` | 知识库根目录；默认交给 `load_settings`（一般为 `w3/sop_knowledge`）。 |
| `-r` / `--recursive` | 是否递归扫描子目录中的 PDF/CSV。 |
| `--rebuild` | 强制重建向量索引（交给 `build_or_load_vectorstore`）。 |
| `--session` | 会话 ID；同一 ID 共享 `InMemoryChatMessageHistory`，不同 ID 互不干扰。 |

#### `main() -> None`

| 项目 | 说明 |
|------|------|
| **作用** | 完整启动流程与终端问答循环。 |
| **参数** | 无。 |
| **返回值** | 无；`quit`/`exit` 退出循环。 |

**内部逻辑要点（顺序）**

1. `load_settings(docs_dir=args.dir, recursive=args.recursive)`：API、路径、检索超参等。
2. `build_or_load_vectorstore`：得到 FAISS 向量库。
3. `chunks_jsonl_path` + `export_vectorstore_documents_to_jsonl` + `load_documents_from_jsonl`：与 w3 一致，为 BM25 提供与向量库一致的语料。
4. `HybridRerankPipeline(vs, bm25_docs, settings)`：混合检索 + Rerank。
5. `build_sop_tools(pipeline, settings)`：得到两个 `BaseTool`。
6. 闭包 `get_session_history(session_id)`：按 `session_id` 返回/创建 `InMemoryChatMessageHistory`。
7. `build_agent_with_history(settings, tools, get_session_history)`：得到带历史的 Runnable。
8. 循环中：`agent.invoke({"input": q}, config={"configurable": {"session_id": session_id}})`，满足 **RunnableWithMessageHistory** 对 `configurable.session_id` 的约定。
9. `/new`：从 `store` 中删除当前 `session_id` 对应的历史，实现「清空本会话记忆」。

**为何这样满足「多轮记忆」**

- LangChain 约定：`RunnableWithMessageHistory` 根据 `config["configurable"]["session_id"]` 取 `BaseChatMessageHistory`，在每次 invoke 前后自动 **读入历史、写入本轮人机消息**。
- `input_messages_key="input"`、`history_messages_key="chat_history"` 与 `agent.py` 里提示词变量名一致。

### 3.4 `agent.py`

**作用**

- 用 **系统提示 + chat_history + 当前用户输入 + agent_scratchpad** 构成 Tool Calling Agent。
- 用 **AgentExecutor** 执行「推理 → 调工具 → 再推理」循环。
- 用 **RunnableWithMessageHistory** 包裹 Executor，实现多轮会话状态。

**常量**

#### `SYSTEM_PROMPT`（`str`）

- 约束模型：**不得凭空编造**表格数或手册条款；必须按场景选择 `sop_document_search` 或 `sop_data_analytics`。
- 强调**指代消解**：追问中的「这个」「上面」需结合 `chat_history` 补全实体后再调工具。

**函数**

#### `build_agent_with_history(settings, tools, get_session_history) -> RunnableWithMessageHistory`

| 参数 | 类型 | 含义 |
|------|------|------|
| `settings` | `Settings` | 来自 `sop_hub.config`，含 `chat_model`、`api_key`、`base_url` 等。 |
| `tools` | `list[BaseTool]` | 一般为 `build_sop_tools` 返回的两个工具。 |
| `get_session_history` | `Callable[[str], BaseChatMessageHistory]` | 入参为 `session_id`，返回该会话的消息存储。 |

| 返回值 | 说明 |
|--------|------|
| `RunnableWithMessageHistory` | 对外表现为：传入 `{"input": "用户一句"}` + `config` 即可；内部自动拼接 `chat_history`。 |

**内部组件说明**

| 组件 | 作用 |
|------|------|
| `ChatOpenAI(...)` | 与 OpenAI 兼容 API 对话；`temperature=0` 降低随机编造。 |
| `ChatPromptTemplate.from_messages([...])` | `chat_history` 占位符对应作业要求的 **MessagesPlaceholder**；`agent_scratchpad` 为 Agent 中间步骤（工具调用与结果）预留。 |
| `create_tool_calling_agent(llm, tools, prompt)` | 生成 **支持工具调用** 的 Runnable Agent（非 ReAct 纯文本）。 |
| `AgentExecutor(..., max_iterations=15, handle_parsing_errors=True)` | 控制最大步数；解析异常时尝试把错误反馈给模型而非直接崩溃。 |
| `callbacks=[ColoredToolCallback()]` | 满足作业「终端可见工具名与入参」。 |
| `RunnableWithMessageHistory(executor, get_session_history, input_messages_key="input", history_messages_key="chat_history", output_messages_key="output")` | Executor 输出为 `{"output": "..."}` 时，把 **output** 写入历史，与官方示例一致。 |

**为何用 `create_tool_calling_agent` 而不是手写 ReAct**

- 与 OpenAI 兼容的 **function calling / tool_calls** 对齐，结构化程度高，国产模型适配通常比纯文本 ReAct 更稳。
- 与 `langchain_experimental` 里 Pandas Agent 的 `agent_type="tool-calling"` 体系一致，便于维护。

**为何用 `RunnableWithMessageHistory` 而不是手写 `ConversationBufferMemory`**

- 作业明确要求 **RunnableWithMessageHistory**；与 LCEL/Runnable 体系一致，会话 ID 可配置，后续可替换为 Redis 等持久化实现。

### 3.5 `tools.py`

**作用**

- 把 w3 中的 **混合检索管线** 与 **Pandas 表格 Agent** 封装成两个 **LangChain Tool**，供大模型通过 **tool calling** 调用。

**函数**

#### `build_sop_tools(pipeline, settings) -> list`（内部返回两个 `BaseTool`）

| 参数 | 类型 | 含义 |
|------|------|------|
| `pipeline` | `HybridRerankPipeline` | 已绑定 FAISS + BM25 + Rerank 的检索实例。 |
| `settings` | `Settings` | 用于 `run_table_agent` 读 CSV、LLM 配置等。 |

| 返回值 | 说明 |
|--------|------|
| `[sop_document_search, sop_data_analytics]` | 两个带 `@tool` 装饰器的可调用对象，名称即模型可见工具名。 |

---

#### 工具一：`sop_document_search(query: str) -> str`

| 项目 | 说明 |
|------|------|
| **作用** | 对「手册/政策/SOP 条文类问题」从向量库检索片段，拼接为带文件与页码信息的上下文文本。 |
| **参数 `query`** | 自然语言检索问句；**docstring 中强调**若用户有指代，模型应把实体补全进 `query`。 |
| **返回** | 成功：检索摘要行 + `format_docs_for_rag` 正文；失败：错误说明或「未检索到」。 |

**实现要点**

- 调用 `pipeline.retrieve(q)`，与 w3 主流程一致（BM25+FAISS 混合 + Rerank）。
- 不在工具内再调 LLM 生成答案，只返回 **材料**；最终表述由上层 Agent 统一完成，避免重复生成与责任分散。

---

#### 工具二：`sop_data_analytics(question: str) -> str`

| 项目 | 说明 |
|------|------|
| **作用** | 对「业务表统计/筛选/聚合」类问题，委托 `sop_hub.table_agent.run_table_agent`。 |
| **参数 `question`** | 自然语言数据分析任务描述。 |
| **返回** | Pandas Agent 的最终输出字符串，或异常说明。 |

**为何单独封装为工具**

- 作业要求：表格逻辑与检索分离；**description** 写清「何时用表、何时用手册」，减少模型误用 CSV 去回答「手册里规定什么」类问题。

---

### 3.6 `callbacks.py`

**作用**

- 继承 `BaseCallbackHandler`，在 **工具开始执行** 与 **工具结束** 时向终端打印信息，对应作业中的 **Action / Observation** 可视化。

**函数与类**

#### `_short(s: str, limit: int = 1800) -> str`

| 项目 | 说明 |
|------|------|
| **作用** | 长文本截断，避免 Observation 刷屏。 |
| **参数** | `s`：原始字符串；`limit`：最大长度。 |
| **返回** | 截断后的字符串。 |

#### `ToolTraceColors`（类）

- 仅存放 ANSI 颜色码（青/黄/绿/灰/复位），用于终端区分 **工具名**、**入参**、**观测**。

#### `ColoredToolCallback(BaseCallbackHandler)`

| 方法 | LangChain 回调时机 | 打印内容 |
|------|-------------------|----------|
| `on_tool_start` | 某个 Tool 开始执行 | `[Action] 工具: <name>`、`入参 (Inputs): ...`（优先 `inputs` 的 JSON，否则 `input_str`） |
| `on_tool_end` | Tool 返回后 | `[Observation] ...`（截断） |

**为何用 Callback 而不是 `verbose=True`**

- `verbose=True` 的格式固定，不便区分颜色与「Action/Observation」标签。
- Callback 只关心工具边界，输出更可控，且与作业「展示入参」要求一致。

**注意**

- Windows 请使用 **Windows Terminal** 或支持 ANSI 的控制台，否则可能看不到颜色（文字仍可见）。

## 四、运行时数据流（简述）

1. 用户输入一句 → `RunnableWithMessageHistory` 注入 `chat_history`。
2. `AgentExecutor` 循环：LLM 可能返回 `tool_calls` → 执行 `sop_document_search` / `sop_data_analytics` → 结果进入 `agent_scratchpad` → 再调 LLM 直到给出最终 `output`。
3. 本轮用户句与最终 `output` 写入当前 `session_id` 的 `InMemoryChatMessageHistory`。
4. 下一轮可引用上一轮内容，配合 **SYSTEM_PROMPT** 与工具 **docstring**，完成「指代追问 + 跨工具」场景。

## 五、如何运行（与依赖）

在**仓库根目录**（与 `w3`、`w4` 同级）执行：

```bash
python -m w4
```

依赖：

- 虚拟环境中已安装根目录 `requirements.txt`（含 `langchain-classic`、`langchain-openai` 等）。
- `w3/.env` 中配置 `api_key`、`base_url` 等（由 `sop_hub.config.load_settings` 读取）。
- `w3/sop_knowledge`、`w3/local_faiss_index` 与 w3 一致；首次或损坏时可使用 `--rebuild`（见 `app.py` 参数）。

**演示用数据（须与 PDF 原文一致，勿口头编造阈值）**

- 主表 **`w3/sop_knowledge/data.csv`**（约 **1000** 行，由 `w3/scripts/generate_data_csv.py` 生成）：字段含预测准确率、缺货率、库存周转天数等。分档阈值以《S&OP **供应链绩效考核**指标管理规范 (v2023)》**§4.3** 为准：**A 优秀 ≥85%**；**B 合格 75%～85%**；**C 不合格 <75%**（「不合格」硬线为 **75%**，不是 85%）。缺货率 **§5.3**、连续两个月升级条件见《异常预警…》**§3.2**（连续两月「不合格」指 **两月均 <75%**）。说明全文见 **`w3/sop_knowledge/DATA_数据设计说明.txt`**；演示话术见 **`w4/DEMO_SOP中文知识库_data.md`**。
- 若仍保留英文主题线月度表 `sop_monthly_kpi_by_kb_article.csv`，可配合 `CSV与三篇PDF对照表.txt` 与 `DEMO_连环追问_知识库三篇.md`（历史脚本）。

## 六、小结：方法选择与作业目标的对应

| 目标 | 方法 | 原因 |
|------|------|------|
| 去掉显式 Router | Tool Calling Agent | 由模型读工具描述与当前问题，动态选工具，比单独分类器更贴近「Agent」定义。 |
| 工具边界清晰 | 两个 `@tool` + 长 docstring | 模型几乎仅凭 description 决策；与课堂强调的「description 要精准」一致。 |
| 多轮指代 | `chat_history` + 系统提示 | 历史进入 prompt，模型可把「这个准确率」展开成完整检索 query。 |
| 可观测性 | `ColoredToolCallback` | 满足「终端看到工具名与入参、思考链路」的展示要求。 |
| 复用 w3 | `import sop_hub` + `w3` 路径注入 | 索引、Rerank、Pandas Agent 全部复用，不重复造轮子。 |
